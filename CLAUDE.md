# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project purpose

`moneta2lezeni` synchronizes incoming transactions from the Moneta bank transparent-account API (Czech Climbing Federation account `246594777`) into a local DB, and posts matching payments to the Lezení registration API. A small React UI surfaces transactions that could not be auto-processed so a human can fix them.

Data flow:

```
MonetaTransparentAccountController.GET /moneta/{acct}
  → SynchronizeTransactionService.syncTransactions
  → MonetaTransparentAccountService.fetchAllTransactions   (paginated REST to Moneta)
  → KafkaTransactionProducer → topic "new-transactions"
  → KafkaTransactionConsumer
      ├─ auto-processable (amount>50, VS in [10000,99999]) → LezeniApiService.callExternalApi → AUTO_PROCESSED / AUTO_FAILED
      ├─ PENDING_MANUAL   → saved, surfaced in UI
      └─ manual fix from UI → re-publish to Kafka → MANUALY_FIXED
  → TransactionRepository (JPA, Postgres)
```

The React UI (`ui-client/`) talks only to `TransactionController` (`/transactions`, `/transactions/{id}`, `/transactions/manual`) and triggers a sync through `MonetaTransparentAccountController` (`/moneta/{accountName}`). Vite dev-server proxies `/api/*` → backend `:8080` (prefix stripped).

## Tech stack & non-obvious choices

- **Java 25 toolchain**, Spring Boot **4.0.1**, Gradle wrapper (`./gradlew` / `gradlew.bat`).
- Resilience4j: official `io.github.resilience4j:resilience4j-spring-boot4:2.4.0` (Maven Central, released March 2026 via PR #2384). Earlier the project pinned a `com.github.rimuln.resilience4j` jitpack fork because upstream Spring Boot 4 support hadn't shipped — that workaround is gone, and the `jitpack.io` repository was removed from `build.gradle`.
- Resilience4j integration points:
  - `@RateLimiter(name = "monetaSync")` on `MonetaTransparentAccountController` (1 call / 10 s, returns 429 via `syncRateLimitFallback`).
  - `@CircuitBreaker(name = "lezeniApi")` on `LezeniApiService.callExternalApi` (fallback returns `PENDING`).
  - Config in `application.yaml` under `resilience4j.*`.
- Kafka runs in **KRaft mode** (no Zookeeper) with two listeners — `INTERNAL` (`kafka:29092`, used inside Docker) and `EXTERNAL` (`localhost:9092`, used when running backend from the IDE).
- JPA uses `ddl-auto: update` (prod) / `create-drop` (test, H2 in PostgreSQL compatibility mode). Entity is `Transaction` → table `chs_transactions`. Optimistic locking via `@Version`.
- QueryDSL is used via `QuerydslPredicateExecutor` (see `TransactionService.findNotProcessed`). `QTransaction` is generated by the annotation processor — rebuild after entity changes.
- MapStruct (`TransactionMapper`) is Spring-wired (`componentModel = "spring"`) and has a Lombok binding (`lombok-mapstruct-binding`) — both annotation processors must stay in `build.gradle`.
- `TransactionDto.equals/hashCode` is restricted to `(transactionNumber, transactionDate)` via `@EqualsAndHashCode(of=…)` — be aware when using DTOs in sets/maps.
- `LEZENI_API_TOKEN` is a **required** env var (injected into the Lezení `RestClient` as `X-Api-Token`). For docker-compose it comes from `.env` at repo root.

## Common commands

Backend (run from repo root):

```bash
./gradlew build                 # full build + tests + jacoco report
./gradlew bootRun               # run the Spring Boot app locally (needs LEZENI_API_TOKEN, local Kafka/Postgres/Redis)
./gradlew test                  # JUnit5 tests; activates spring profile "test" (H2)
./gradlew test --tests "navrat.name.moneta2lezeni.service.MonetaTransparentAccountServiceTest"
./gradlew test --tests "*ConsumerTest.consume_*"
./gradlew jacocoTestReport      # also runs after `test` automatically (test.finalizedBy)
./gradlew bootJar               # produces build/libs/*.jar (used by Dockerfile)
./gradlew rewriteRun            # OpenRewrite recipes (Spring Boot 4 upgrade, static analysis, tab normalization)
```

SonarQube analysis (Windows cmd — expects a local Sonar at `http://localhost:9000`):

```cmd
sonarqube-analyze.cmd
```

This script sources `setup-sonar-env.cmd` for `SONAR_TOKEN`, runs `clean build test jacocoTestReport`, runs the frontend coverage task in Docker, then `gradlew sonar`. `setup-sonar-env.cmd` is **untracked and contains a real token** — keep it out of git (it is not in `.gitignore` today; don't commit it).

Frontend (`ui-client/`):

```bash
npm install
npm run dev                     # Vite on :3000, proxies /api → VITE_API_URL (default http://localhost:8080)
npm run build
npm run lint                    # ESLint flat config (eslint.config.js)
npm test                        # Vitest; test files live in ui-client/tests/**/*.test.{ts,tsx}
npm run test:coverage
```

Docker stack (`docker-compose.yaml` spins up postgres, kafka-KRaft, redis, backend, frontend):

```bash
docker compose up               # full stack
```

`docker-compose.override.yml` is gitignored and in this tree it **no-ops the `backend` service** (`entrypoint: exit 0`) — the intent is "I'm running the backend from the IDE, only bring up infra + frontend". Delete or edit the override if you actually want the backend container.

## Environment & config

- `.env` (gitignored) — holds `LEZENI_API_TOKEN` for docker-compose.
- `application.yaml` reads `SPRING_DATASOURCE_URL`, `SPRING_KAFKA_BOOTSTRAP_SERVERS`, `SPRING_REDIS_HOST`, `LEZENI_API_TOKEN` from env, with sensible localhost defaults.
- `application-test.yaml` (H2 + fake endpoints) is auto-activated because `tasks.named('test')` sets `spring.profiles.active=test`.
- Actuator endpoints exposed: `health,circuitbreakers,ratelimiters,metrics`.
- CORS is hard-coded to `http://localhost:3000` on both controllers — update both if the UI origin changes.

## Testing notes

- Tests use JUnit 5 (`useJUnitPlatform()`), H2 in PostgreSQL mode, and `spring-boot-starter-*-test` slices. `jvmArgs("-XX:+EnableDynamicAgentLoading")` is set so Mockito/inline agents work on JDK 25.
- `src/test/java/.../utils/` holds shared test helpers; existing suites cover service/consumer/controller/repository layers and a `Resilence4JConfigTest` that exercises the annotations end-to-end.
- Frontend tests use Vitest + jsdom + `@testing-library/react`; setup in `ui-client/tests/setup.ts`. Only the setup file exists today — any new tests go under `ui-client/tests/`.

## Known open bug

`bugfix-moneta-pagination.md` (at repo root) documents an unfixed duplicate-detection bug in `MonetaTransparentAccountService.fetchAllTransactions` — the filter matches on `(transactionNumber, transactionSentDate)` which is **not** unique across days (Moneta resets `transactionNumber` daily). The file spells out the fix (add `transactionDate` to the predicate, or compare by `id`) and the data-repair options. Read it before touching the pagination/dedup logic.
